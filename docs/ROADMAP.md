## 1. Анализ текущего проекта

### 1.1 Текущий стек и структура

| Компонент | Реализация |
|-----------|------------|
| **Frontend** | Vanilla JavaScript, один HTML (index.html), hash-роутинг |
| **Данные** | Статические JSON: `data/<class>/<subject>/paragraphs.json`, `quiz-*.json` |
| **Загрузка** | `fetch` + in-memory кэш (Api), подмена через dataProvider из localStorage |
| **Админка** | Отдельная страница (admin.html), без сервера, сохранение в localStorage |
| **Тесты** | Jest, unit-тесты для quizEngine (normalizeAnswer, validateAnswer, shuffle и т.д.) |

### 1.2 Иерархия контента

```
CONFIG (data.js + merge из localStorage)
  └── classes[classId]
        └── subjects[subjectId]  { name, path }
              └── path → data/<class>/<subject>/
                    ├── paragraphs.json  → массив параграфов
                    └── quiz-*.json      → { title, questions[] }
                          └── questions[]  → type: choice | input, q, a/c или answer(s), hint, explanation
```

Параграф: `id`, `title`, `summary`, `sections[]`, `dates[]`, `terms[]`, `people[]`, `quizzes[]` (ссылки на файлы тестов).

### 1.3 Реализованный функционал

- Просмотр параграфов (разделы, даты, понятия, персоны), краткое содержание, разметка (переносы, **жирный**, *курсив*).
- Прохождение тестов: choice (один правильный вариант), input (строка/массив ответов), 3 попытки, пропуск, подсказки.
- «Свой тест»: выбор параграфов → объединение вопросов → до 20 случайных.
- Навигация: hash-URL (#/class/6/english, #/class/6/english/paragraph/19, #/class/6/english/quiz/...), сохранение при F5.
- Админка: классы, предметы, параграфы, тесты; редактирование в памяти + сохранение в localStorage; экспорт/импорт JSON.

### 1.4 Ограничения и риски

| Область | Проблема |
|---------|----------|
| **Масштабируемость** | Всё в JSON и localStorage; нет разделения по пользователям, нет версионирования контента. |
| **Пользователи** | Нет ролей, авторизации, персональной статистики, истории, достижений. |
| **Контент** | Нет изображений в вопросах, нет drag-and-drop в редакторе, генерация тестов только «из параграфов». |
| **Безопасность** | Админка локальная; при переносе в прод потребуется auth, RBAC, валидация на бэкенде. |
| **Поддержка** | Рост vanilla JS усложнит поддержку; нет типов, нет единого состояния. |
| **Данные** | Нет БД → нет транзакций, бэкапов, аналитики, аудита. |

## 2. Предлагаемая архитектура

### 2.1 Нужно ли переходить на фреймворк (React / Vue / Svelte / Next.js)?

**Рекомендация: да.**

- Появляются роли (ученик, учитель, админ), много экранов (кабинет, редакторы, отчёты), сложное состояние (тесты, прогресс, рейтинги).
- Vanilla JS приведёт к дублированию логики, сложной синхронизации URL/состояния и тяжёлому рефакторингу.
- **Предложение:** **React** (или Vue) + **TypeScript** для типизации контрактов с API и моделей данных. Next.js — опционально для SSR/SEO; для внутреннего портала достаточно SPA (Vite + React).

### 2.2 Нужен ли backend (например Node.js)?

**Рекомендация: да.**

- Учёт пользователей, роли, хранение результатов, контент по классам/предметам, генерация тестов по датам/понятиям — требуют серверной логики и БД.
- Без бэкенда невозможно: единая статистика, рейтинги, достижения, экспорт для учителей, безопасная админка.
- **Предложение:** **Node.js** (Express или Fastify) или альтернатива на другом языке (Go, .NET) при наличии экспертизы; для стека JS универсально — **Node.js + Express/Fastify**.

### 2.3 Где хранить данные: JSON или база данных?

**Рекомендация: база данных для всего операционного контента и пользователей; JSON — только как формат импорта/экспорта или начальная загрузка.**

| Данные | Хранение | Обоснование |
|--------|----------|-------------|
| Пользователи, роли, сессии | БД | Безопасность, масштаб, аудит. |
| Классы, предметы, параграфы, тесты, вопросы | БД | Версионирование, права, поиск, связки (даты/понятия → вопросы). |
| Результаты тестов, история, баллы, достижения | БД | Аналитика, рейтинги, отчёты. |
| Медиа (изображения в вопросах) | Файловое хранилище (S3/локальное) + ссылки в БД | Масштаб и CDN. |
| Экспорт/бэкапы | JSON/файлы | Обмен между инстансами, резервные копии. |

### 2.4 Вывод по архитектуре

- **Frontend:** SPA на React (или Vue) + TypeScript, роутинг (React Router), состояние (React Query + Context/Zustand по необходимости).
- **Backend:** REST API (или GraphQL при росте сложности запросов), Node.js + Express/Fastify.
- **База данных:** Реляционная СУБД (PostgreSQL или MySQL) для пользователей, контента, результатов; при необходимости кэш (Redis) для сессий и рейтингов.
- **Файлы:** Хранилище для картинок (локальная папка / S3-совместимое), URL в БД.

## 3. Предлагаемый стек технологий

| Слой | Технологии | Комментарий |
|------|------------|-------------|
| **Frontend** | React 18+, TypeScript, Vite, React Router, React Query (TanStack Query), UI (MUI / Ant Design / Radix + Tailwind) | Типизация, быстрая сборка, удобный запрос к API. |
| **Backend** | Node.js 20+, Express или Fastify, TypeScript | Единый язык, переиспользование типов (shared types). |
| **Database** | PostgreSQL 15+ (или MySQL 8+) | Пользователи, контент, результаты, индексы, JSON-поля при необходимости. |
| **Auth** | JWT (access + refresh) или сессии в Redis; bcrypt для паролей; OAuth2 (Google/МЭШ) — опционально | Роли: student, teacher, admin. |
| **Storage** | Локальная папка `uploads/` или S3-совместимое (MinIO, AWS S3) | Изображения в вопросах и параграфах. |
| **Deploy** | Docker (front + api + db); Nginx как reverse proxy; опционально CI/CD (GitHub Actions) | Воспроизводимая среда, масштабирование. |

Дополнительно:

- **Тестирование:** Jest (unit), React Testing Library (компоненты), Playwright/Cypress (e2e), Supertest (API).
- **Линтинг/форматирование:** ESLint, Prettier.
- **Мониторинг:** логирование (Winston/Pino), при росте — метрики и алерты.

## 5. Архитектура базы данных (концептуальная схема)

### 5.1 Основные сущности

- **users** — id, email, password_hash, role (student | teacher | admin), full_name, class_id (FK, для учеников), created_at, updated_at.
- **classes** — id, name, sort_order.
- **subjects** — id, class_id (FK), name, path_slug (уникальный в рамках класса).
- **paragraphs** — id, subject_id (FK), external_id (например §17), title, summary, sort_order, created_at, updated_at.
- **paragraph_sections** — id, paragraph_id (FK), title, content, sort_order.
- **paragraph_dates** — id, paragraph_id (FK), year, event.
- **paragraph_terms** — id, paragraph_id (FK), term, definition.
- **paragraph_people** — id, paragraph_id (FK), name, info.
- **quizzes** — id, subject_id (FK), title, file_slug (или убрать, если всё в БД), created_at, updated_at.
- **quiz_questions** — id, quiz_id (FK), type (choice | input), question_text, options (JSON: варианты для choice), correct_index (choice), correct_answers (JSON для input), hint, explanation, image_url (nullable), sort_order.
- **quiz_paragraph_links** — quiz_id, paragraph_id (связь теста с параграфами для «теста по параграфу»).
- **test_attempts** — id, user_id (FK), quiz_id (FK), started_at, finished_at, score, max_score, status (in_progress | completed | abandoned).
- **test_attempt_answers** — id, attempt_id (FK), question_id (FK), user_answer (JSON/text), is_correct, attempt_number.
- **achievements** — id, code, name, description, icon_url, condition (JSON или отдельная таблица правил).
- **user_achievements** — user_id, achievement_id, unlocked_at.
- **school_classes** — привязка учеников к классам (если класс = учебная группа): id, name; users.class_id → school_classes.id.

Для генерации тестов «только по датам» / «только по понятиям» можно завести отдельные сущности (например, вопросы, привязанные к paragraph_dates или paragraph_terms) или флаги/типы у quiz_questions.

### 5.2 Связи (кратко)

- Класс → много предметов; предмет → много параграфов и много тестов.
- Параграф → разделы, даты, термины, персоны; тест → много вопросов.
- Пользователь → много попыток тестов; попытка → много ответов.
- Достижения выдаются по правилам на основе попыток и ответов.

## 6. Основные модули системы

| Модуль | Назначение | Основные функции |
|--------|------------|------------------|
| **Auth** | Регистрация, вход, JWT, роли | login, register, refresh, logout, middleware RBAC. |
| **Users** | Профили, привязка к классу | CRUD профиля, смена пароля, список учеников класса (для учителя). |
| **Classes & Subjects** | Иерархия контента | CRUD классов и предметов (админ/учитель). |
| **Paragraphs** | Темы, разделы, даты, термины, персоны | CRUD параграфов и подсущностей; поддержка разметки и изображений. |
| **Quizzes** | Тесты и вопросы | CRUD тестов, CRUD вопросов (в т.ч. изображения), порядок (drag-and-drop через sort_order). |
| **Quiz Engine** | Генерация вариантов теста | По параграфам; по типу (только даты / только понятия); лимит вопросов, рандомизация. |
| **Test Passing** | Прохождение теста | Старт попытки, отправка ответов, завершение, подсчёт баллов. |
| **Results & Statistics** | Результаты и отчёты | История попыток, баллы по ученику/классу/тесту, экспорт (CSV/Excel). |
| **Achievements & Ratings** | Геймификация | Правила выдачи достижений; рейтинг по баллам/количеству тестов. |
| **Admin** | Управление контентом и пользователями | Управление классами, предметами, параграфами, тестами; роли; при необходимости модерация. |
| **Storage** | Загрузка файлов | Загрузка изображений к вопросам/параграфам, отдача по URL. |

## 7. Roadmap разработки

Разбивка по спринтам (условно 2 недели на спринт). Задачи — без реализации, только план.

---

### Sprint 1 — Фундамент и инфраструктура

**Цель:** Поднять backend и БД, заложить контур API и авторизации, подготовить фронт к миграции.

**Задачи:**
- Инициализация репозитория: monorepo или раздельные frontend/backend; общие типы (packages/shared-types или папка в api).
- Настройка БД: PostgreSQL (локально/Docker), миграции (например node-pg-migrate или TypeORM migrations).
- Базовая схема: users, roles, classes, subjects (минимальные поля).
- Backend: Express/Fastify, роуты health, версия API; middleware логирования и обработки ошибок.
- Регистрация и вход: хэширование паролей (bcrypt), выдача JWT (access + refresh), middleware проверки токена и роли.
- Frontend: настройка Vite + React + TypeScript, React Router, базовый layout (шапка, меню по ролям), экраны Login/Register (заглушки), вызов API авторизации и сохранение токена.

**Результат:** Пользователь может зарегистрироваться и войти; backend отдаёт JWT; фронт показывает разный layout в зависимости от роли (ученик/учитель/админ); БД с пользователями и ролями.

---

### Sprint 2 — Контент: классы, предметы, параграфы (API + админ)

**Цель:** Перенести контент из JSON в БД и обеспечить его управление через API и админку.

**Задачи:**
- Миграции: paragraphs, paragraph_sections, paragraph_dates, paragraph_terms, paragraph_people; связи с subjects.
- API: CRUD для классов и предметов (с проверкой роли admin/teacher).
- API: CRUD для параграфов (включая разделы, даты, термины, персоны); валидация и санитизация на бэкенде.
- Скрипт или endpoint импорта из текущих JSON в БД (одноразовая миграция контента).
- Frontend (админ/учитель): экраны «Классы», «Предметы», «Параграфы» — список, создание, редактирование, удаление; вызовы API.
- Роутинг и права: защита роутов по роли; редирект неавторизованных на логин.

**Результат:** Весь текущий контент (классы, предметы, параграфы) живёт в БД; админ/учитель управляет им через веб-интерфейс; ученик пока может видеть контент в режиме только чтения (если реализовать публичные GET-эндпоинты).

---

### Sprint 3 — Тесты и вопросы в БД, редактор тестов

**Цель:** Тесты и вопросы в БД; редактор тестов с drag-and-drop и поддержкой изображений.

**Задачи:**
- Миграции: quizzes, quiz_questions, при необходимости quiz_paragraph_links; поле image_url у вопросов.
- API: CRUD тестов и вопросов; загрузка изображения к вопросу (multer + сохранение в storage), отдача URL.
- Логика генерации теста: по списку параграфов (как сейчас «свой тест»); задел под фильтр «только даты» / «только понятия» (флаг или тип вопроса).
- Frontend: редактор теста — список вопросов, добавление/редактирование/удаление; поддержка type choice/input; загрузка картинки; drag-and-drop для порядка (sort_order).
- Отображение вопросов с картинкой на фронте при прохождении теста (пока без сохранения результата).

**Результат:** Тесты и вопросы хранятся в БД; учитель/админ создаёт и редактирует тесты в удобном редакторе; в вопросах можно использовать изображения; генерация теста по параграфам реализована на API.

---

### Sprint 4 — Прохождение теста и сохранение результатов

**Цель:** Учёт попыток, сохранение ответов, подсчёт баллов и отображение результата.

**Задачи:**
- Миграции: test_attempts, test_attempt_answers.
- API: старт попытки (POST), отправка ответа на вопрос (POST), завершение попытки (POST); подсчёт score/max_score на бэкенде.
- Валидация: только свой класс/доступ к тесту; одна активная попытка или политика повторных прохождений.
- Frontend: экран прохождения теста — таймер/прогресс, отправка ответов на API, переход к следующему вопросу, завершение и экран результата (баллы, правильные/неправильные).
- Сохранение состояния при обновлении страницы (опционально: восстановление активной попытки по attempt_id).

**Результат:** Ученик проходит тест; ответы и баллы сохраняются в БД; по завершении видит результат; учитель/админ может видеть попытки в списке (базово).

---

### Sprint 5 — Статистика, история, рейтинг и достижения

**Цель:** Личный кабинет ученика, история тестов, рейтинг, система достижений.

**Задачи:**
- API: история попыток ученика (по пользователю, с фильтрами); агрегаты по классу/тесту для учителя.
- API: рейтинг (топ по баллам/количеству тестов за период); по классу и глобальный.
- Таблицы и API достижений: справочник achievements, user_achievements; правила выдачи (например «пройти 5 тестов», «набрать 100% в тесте»).
- Frontend: кабинет ученика — история тестов, баллы, достижения, место в рейтинге.
- Frontend: для учителя — список учеников класса, сводка по тестам, экспорт в CSV/Excel (API + кнопка).

**Результат:** Ученик видит свою историю и достижения; рейтинг отображается; учитель видит статистику по классу и может экспортировать результаты.

---

### Sprint 6 — Генерация тестов по датам/понятиям и доработки

**Цель:** Гибкая генерация тестов (по датам, по понятиям); доработка UX и безопасности.

**Задачи:**
- Модель: привязка вопросов к типу контента (дата/термин/раздел) или отдельные сущности «вопрос по дате»/«вопрос по понятию».
- API: генерация теста с фильтрами — только даты, только понятия, только персоны, комбинированный; лимит вопросов, рандомизация.
- Frontend: учитель выбирает параграф(ы) и тип генерации (даты/понятия/все), получает сгенерированный тест (создание черновика теста в БД или одноразовая выдача).
- Безопасность: rate limiting, CORS, санитизация входов; аудит критичных действий (логирование).
- Доработки: сброс пароля (email или по запросу админа), базовая валидация прав на уровне API для всех ресурсов.

**Результат:** Учитель может создавать тесты «только по датам» или «только по понятиям»; безопасность и аудит усилены; сброс пароля доступен.

---

### Sprint 7+ — Масштабирование и полировка

**Цель:** Стабильность, производительность, расширение функционала.

**Задачи (примеры):**
- Изображения в параграфах (разделы, блоки); галерея в админке.
- Кэширование: Redis для сессий или горячих данных рейтинга; кэш публичного контента (параграфы по предмету).
- E2E-тесты: сценарии входа, прохождения теста, создания теста учителем.
- CI/CD: сборка, тесты, деплой в тестовое/прод окружение (Docker).
- Мониторинг: логи, метрики (время ответа API, ошибки); уведомления при падении сервисов.
- Документация API (OpenAPI/Swagger), краткая документация для учителей и админов.

**Результат:** Система готова к эксплуатации в школе; контент и пользователи масштабируются; поддержка и развитие упрощены.

## 8. Риски и зависимости

| Риск | Митигация |
|------|-----------|
| Миграция контента из JSON в БД с ошибками | Скрипт импорта с валидацией и откатом; бэкап БД до миграции. |
| Рост сложности фронта при добавлении ролей | Чёткое разделение features (student, teacher, admin), общие компоненты в shared. |
| Производительность при большом числе попыток | Индексы по user_id, quiz_id, started_at; пагинация и агрегаты в БД. |
| Безопасность (инъекции, XSS) | Валидация и санитизация на backend; на фронте — экранирование и Content-Security-Policy. |

## 9. Итоговая сводка решений

| Вопрос | Решение |
|--------|---------|
| Фреймворк | **React + TypeScript** (Vite); при необходимости Next.js для SSR. |

| Хранение данных | **PostgreSQL** для всего операционного контента и пользователей; **файловое хранилище** для изображений; JSON — импорт/экспорт. |
