# Roadmap разработки — School Portal

**Версия:** 2.0  
**Подход:** Два этапа (Stage 1 — локальный портал, Stage 2 — полноценная платформа)  
**Статус:** Планирование, без реализации

---

## 1. Концепция проекта

### 1.1 Текущее состояние (прототип)

Рабочий прототип учебного тестового портала на **чистом JavaScript** и **JSON**:

- **Иерархия:** классы → предметы → параграфы → тесты → вопросы.
- **Типы вопросов:** choice (выбор варианта), input (ввод ответа).
- **Функции:** просмотр параграфов, прохождение тестов, «свой тест» из выбранных параграфов (до 20 случайных вопросов).
- **Админка:** отдельная страница, без сервера; данные в localStorage + экспорт/импорт JSON.
- **Навигация:** hash-URL, сохранение при обновлении страницы.

### 1.2 Целевое видение

**Этап 1 — Домашний портал (один администратор):**  
Мощный и удобный офлайн-инструмент для учителя или родителя: один человек управляет контентом, ученики (дома или в классе) проходят тесты и карточки без учёток. Всё на статике, JSON и localStorage.

**Этап 2 — Образовательная платформа:**  
Многопользовательская система с учётами, ролями, статистикой, классами и заданиями. Backend, БД, полный функционал для школы.

---

## 2. Два этапа разработки

| Критерий | Stage 1 — Локальный домашний портал | Stage 2 — Полноценная платформа |
|----------|-------------------------------------|----------------------------------|
| **Цель** | Удобный и мощный инструмент для одного админа (учитель/родитель) | Многопользовательская образовательная платформа |
| **Backend** | Нет | Да (Node.js и т.п.) |
| **База данных** | Нет | PostgreSQL (или аналог) |
| **Учётные записи** | Нет | Ученики, учителя, админы |
| **Рейтинги / статистика** | Нет (или только локальная в localStorage) | Да, по пользователям и классам |
| **Аутентификация** | Нет | Да |
| **Хранение** | Статика, JSON, localStorage | БД + файловое хранилище |
| **Результат** | Офлайн-инструмент, можно открывать с диска или с любого хостинга статики | Онлайн-платформа с ролями и аналитикой |

## 3. Архитектура Stage 1 (Локальный домашний портал)

### 3.1 Ограничения и принципы

- **Без сервера:** только статические файлы (HTML, CSS, JS, JSON, изображения).
- **Данные:** единый формат JSON для всех предметов; при необходимости — один «бандл» (например `data.json`) или разбиение по путям `data/<class>/<subject>/`.
- **Персистентность:** localStorage для правок админа и опционально для локальной статистики/прогресса; экспорт/импорт JSON для переноса и бэкапов.
- **Офлайн:** возможность работать без интернета (PWA или просто открытие с диска).

### 3.2 Универсальная структура JSON (целевая)

Единая схема для любого предмета (история, английский, математика и т.д.):

```json
{
  "version": 1,
  "meta": { "classId": "6", "subjectId": "history", "title": "История", "updated": "2025-03-01" },
  "paragraphs": [
    {
      "id": "p-17",
      "title": "§17. Название темы",
      "summary": "Краткое описание.",
      "image": null,
      "sections": [
        { "title": "Подзаголовок", "content": "Текст.", "image": null }
      ],
      "dates": [
        { "id": "d-1", "year": "1253", "event": "Событие", "image": null }
      ],
      "terms": [
        { "id": "t-1", "term": "Термин", "definition": "Определение", "image": null }
      ],
      "people": [
        { "id": "h-1", "name": "Имя", "info": "Инфо", "image": null }
      ],
      "quizzes": [
        { "id": "q-17", "title": "Тест по §17", "source": "manual" }
      ]
    }
  ],
  "quizzes": [
    {
      "id": "q-17",
      "title": "Тест по §17",
      "source": "manual",
      "difficulty": "medium",
      "questions": [ ... ]
    }
  ]
}
```

- **version** — версия схемы для миграций и валидации.
- **Поля image** — ссылки на файлы (относительные пути или base64 для офлайн); единообразие для параграфов, разделов, дат, терминов, персон и вопросов.
- **source** у теста: `manual` (создан вручную), `dates` | `terms` | `people` | `paragraphs` (автогенерация).
- **id** у дат/терминов/персон — для привязки сгенерированных вопросов и сложности.

Это **целевая** структура; текущие файлы можно постепенно приводить к ней (обратная совместимость через адаптеры).

### 3.3 Модульная архитектура движка тестов

- **Ядро (quiz engine):** чистая логика: нормализация ответа, проверка (choice/input), лимит попыток, пропуск. Без DOM.
- **Генераторы тестов:** отдельные модули:
  - из параграфов (текущая логика);
  - из дат (дата → вопрос «год/событие»);
  - из терминов (термин → вопрос «определение/термин»);
  - из персон (имя → инфо);
  - комбинированный (смесь типов).
- **Адаптивная сложность:** у каждого вопроса или шаблона — уровень сложности (easy/medium/hard); при генерации или при «своём тесте» пользователь или система выбирает уровень; при необходимости — учёт прошлых ошибок из localStorage для подбора следующего теста (опционально в Stage 1).

### 3.5 Рекомендуемые библиотеки (Stage 1)

| Задача | Библиотека | Зачем |
|--------|------------|--------|
| Сборка/минификация (опционально) | Vite или esbuild | Быстрая сборка, один бандл для продакшена |
| Drag & Drop в админке | SortableJS (или нативный HTML5 DnD) | Лёгкий, без фреймворка |
| Валидация JSON-схемы | Ajv (опционально) | Проверка контента при импорте и в админке |
| Работа с датами (если нужно) | date-fns или dayjs | Лёгкие, без монстров |
| Локальное хранилище | Нативный localStorage + обёртка | Простота, без зависимостей |

Фреймворки (React/Vue) в Stage 1 **не обязательны**: текущий vanilla JS допустим при чётком разделении модулей (engine, generators, admin, UI). Если команда планирует быстрый переход на Stage 2 — можно сразу ввести TypeScript и модульную структуру под будущий перенос.

## 4. Архитектура Stage 2 (Полноценная платформа)

### 4.1 Компоненты

- **Frontend:** SPA на React (или Vue) + TypeScript; роутинг, роли (ученик, учитель, админ).
- **Backend:** Node.js (Express/Fastify), REST API; аутентификация (JWT или сессии), авторизация по ролям.
- **БД:** PostgreSQL (или MySQL): пользователи, классы, предметы, параграфы, тесты, вопросы, попытки, достижения.
- **Файлы:** изображения — локальная папка или S3-совместимое хранилище; ссылки в БД.
- **Импорт:** контент из Stage 1 (JSON) загружается в БД через скрипт/эндпоинт миграции.

### 4.2 Новый функционал (без детальной реализации)

- Учётные записи: регистрация, вход, сброс пароля.
- Роли: ученик, учитель, администратор.
- Профили учеников, привязка к школьным классам.
- Кабинет учителя: создание/редактирование контента, назначение тестов, просмотр статистики по классу/ученику, экспорт результатов.
- Статистика и прогресс: история прохождения, баллы, прогресс по темам.
- Рейтинг/лидерборд: по классу, по школе.
- Назначения (assignments): учитель назначает тест классу или ученику с дедлайном.
- Аналитика тестов: какие вопросы чаще ошибочные, время прохождения.

### 4.3 Структура проекта (Stage 2, целевая)

```
school-portal/
├── apps/
│   ├── web/                 # SPA (React + Vite + TypeScript)

│   │       ├── app/

│   │       ├── features/    # Тесты, параграфы, карточки, статистика
│   │       ├── shared/      # API-клиент, типы, UI-кит
│   │       └── ...
│   └── api/                 # Backend (Node + Express/Fastify)
│       └── src/
│           ├── modules/     # auth, users, classes, subjects, paragraphs, quizzes, attempts, achievements
│           ├── migrations/

├── packages/

```

## 5. Дорожная карта по спринтам

Задачи упорядочены **от простых к более сложным** в рамках каждого этапа.

---

### Этап 1 — Локальный домашний портал (Single Admin)

---

#### Спринт 1.1 — Универсальная схема JSON и импорт

**Цель:** Единый контракт данных для всех предметов и безболезненное расширение.

**Задачи:**
- Описать целевую JSON-схему (параграфы, разделы, даты, термины, персоны, тесты, вопросы) с полями `id`, `version`, `image` где нужно.
- Реализовать адаптер: текущие `paragraphs.json` + `quiz-*.json` → единый формат в памяти (без обязательного переписывания файлов на диск в первом спринте).
- Добавить загрузку либо одного `content.json` по предмету, либо текущей пары paragraphs + quiz-файлов с маппингом в общую модель.
- В админке при экспорте сохранять уже новый формат (если редактировали через админку).

**Результат:** Приложение и админка работают с унифицированной моделью; контент можно хранить как один файл по предмету или как сейчас — два типа файлов.

---

#### Спринт 1.2 — Поддержка изображений в параграфах и вопросах

**Цель:** Картинки в разделах параграфа и в вопросах тестов.

**Задачи:**
- В схеме параграфа и вопроса зарезервировать поле `image` (URL или относительный путь; опционально base64 для офлайн).
- В UI параграфа выводить изображение раздела/блока, если указано.
- В движке теста и в UI вопроса выводить изображение перед текстом вопроса.
- В админке: загрузка файла изображения → сохранение в `data/<class>/<subject>/media/` и запись пути в JSON (или base64 в localStorage при редактировании с последующим экспортом в файл).
- Ограничение размера/формата (например, до 500 KB, jpeg/png/webp) на клиенте.

**Результат:** В параграфах и в вопросах отображаются изображения; админ может загружать их через панель без сервера (файлы рядом с данными или в экспорте).

---

#### Спринт 1.3 — Генерация тестов по датам, терминам и персонaм

**Цель:** Автоматические тесты из уже имеющихся данных параграфа.

**Задачи:**
- Модуль «генератор из дат»: по выбранным параграфам собрать все `dates`; генерировать вопросы типа «год по событию» / «событие по году» (choice или input).
- Модуль «генератор из терминов»: по `terms` — «термин по определению» / «определение по термину».
- Модуль «генератор из персон»: по `people` — «имя по описанию» / «описание по имени».
- В UI: выбор типа генерации (по параграфам / только даты / только термины / только персоны) и объём (все выбранные параграфы или лимит).
- Сохранение сгенерированного теста в памяти и в localStorage как отдельный тест с `source: "dates" | "terms" | "people"` и последующий экспорт в JSON.

**Результат:** Учитель/родитель может одним действием получить тест только по датам, только по понятиям или только по персонам и проходить его как обычный тест.

---

#### Спринт 1.4 — Адаптивная сложность тестов

**Цель:** Уровни сложности и возможность подстраивать тест под ученика.

**Задачи:**
- В схеме вопроса (и при генерации) добавить поле `difficulty`: `easy` | `medium` | `hard`.
- При ручном создании теста в админке — выбор сложности для каждого вопроса.
- В генераторах (даты, термины, персоны) задавать сложность по правилам (например, короткие формулировки — easy, длинные — hard) или по умолчанию medium.
- В UI «собственного теста» и при выборе готового теста — фильтр по сложности (только easy, mixed, только hard) или слайдер.
- Опционально: в localStorage сохранять последние ошибки по questionId; при следующем запуске «своего теста» чуть чаще подмешивать сложные вопросы, где были ошибки (упрощённая адаптация).

**Результат:** Тесты и вопросы имеют уровень сложности; пользователь может выбирать сложность при формировании теста; задел под простую адаптацию через localStorage.

---

#### Спринт 1.5 — Карточки (Flashcards)

**Цель:** Режим запоминания «карточка — ответ».

**Задачи:**
- Модель: «колода» = выбранные параграфы или выбранные типы (даты, термины, персоны); одна карточка = один элемент (одна дата, один термин, одна персона).
- UI: показ «лицевой» стороны (например, термин или год), по клику — «оборот» (определение, событие, инфо о персоне); листание вперёд/назад, перемешивание.
- Переиспользование данных из параграфов (dates, terms, people); при наличии изображений — показ на карточке.
- Опционально: сохранение в localStorage прогресса «просмотрено / знаю / не знаю» для последующей фильтрации карточек (повтор только «не знаю»).

**Результат:** Режим карточек по датам, терминам и персонам из выбранных параграфов; можно использовать офлайн.

---

#### Спринт 1.6 — Drag & Drop в админке и улучшение редактора тестов

**Цель:** Удобное упорядочивание без сервера.

**Задачи:**
- В админке: порядок разделов параграфа, дат, терминов, персон — перетаскивание (SortableJS или нативный DnD).
- Порядок вопросов в тесте — drag & drop; сохранение `order` или переиндексация в массиве при сохранении.
- Визуальная обратная связь при перетаскивании (подсветка зоны, placeholder).
- Сохранение порядка в JSON (и в localStorage); экспорт с сохранённым порядком.

**Результат:** Админ выстраивает порядок разделов и вопросов перетаскиванием; данные остаются в JSON/localStorage.

---

#### Спринт 1.7 — Улучшение UI и темы

**Цель:** Современный и приятный интерфейс.

**Задачи:**
- Обновить типографику, отступы, контраст (доступность).
- Единые компоненты: кнопки, карточки, поля ввода, алерты (можно на CSS-переменных без фреймворка).
- Адаптивная вёрстка: комфортное использование на планшете и телефоне (списки параграфов, прохождение теста, карточки).
- Опционально: переключатель темы (светлая/тёмная) через CSS-переменные и сохранение выбора в localStorage.

**Результат:** Единый современный вид портала и админки; удобство на разных устройствах.

---

#### Спринт 1.8 — Лёгкая админка и экспорт/импорт

**Цель:** Стабильное управление контентом без сервера.

**Задачи:**
- Полный цикл: создание/редактирование класса, предмета, параграфа, теста с нуля в админке; все данные уходят в единую модель и в localStorage.
- Экспорт: один JSON-файл по предмету (или один общий архив) в целевом формате схемы; включение медиа как base64 или отдельная папка в архиве (ZIP через JS-библиотеку).
- Импорт: загрузка JSON (и при необходимости распаковка ZIP); валидация по схеме (Ajv или ручная); подмена/merge в localStorage и перезагрузка списков.
- Краткая справка в админке: «как экспортировать бэкап», «как перенести на другой компьютер».

**Результат:** Админ может вести весь контент в браузере и переносить его экспортом/импортом без потери данных.

---

#### Спринт 1.9 — Модульность движка тестов и PWA (опционально)

**Цель:** Чистая архитектура движка и возможность офлайн.

**Задачи:**
- Вынести генераторы (fromDates, fromTerms, fromPeople, fromParagraphs) в отдельные модули с единым интерфейсом: `generate(sourceConfig) → { questions, meta }`.
- Движок тестов только принимает массив вопросов и настройки (лимит попыток, показ подсказок); без привязки к источнику.
- Опционально: PWA — manifest.json, service worker для кэширования статики и JSON; иконки; «Установить» в браузер.
- Тесты (Jest): покрытие генераторов (корректное число вопросов, типы choice/input) и движка (проверка ответов, пропуск).

**Результат:** Движок и генераторы разделены; приложение можно использовать офлайн как PWA.

### Этап 2 — Полноценная образовательная платформа

(Кратко; детализация — после стабилизации Stage 1.)

---

#### Спринт 2.1 — Инфраструктура: backend, БД, auth

**Цель:** Работающий API и учёт пользователей.

**Задачи:**
- Поднять backend (Node.js + Express/Fastify), PostgreSQL, миграции (users, roles, classes, subjects, paragraphs, quizzes, questions).
- Регистрация, вход, JWT (access + refresh), роли student | teacher | admin.
- Frontend: экраны Login/Register, сохранение токена, защита роутов по роли.

**Результат:** Пользователи могут регистрироваться и входить; API отдаёт JWT; фронт различает роли.

---

#### Спринт 2.2 — Миграция контента Stage 1 → БД

**Цель:** Перенести JSON и медиа в платформу.

**Задачи:**
- Скрипт/эндпоинт импорта: чтение JSON (и медиа) из формата Stage 1 → заполнение таблиц (classes, subjects, paragraphs, sections, dates, terms, people, quizzes, questions).
- Валидация и обработка дубликатов (по id или по естественному ключу).
- Загрузка изображений в хранилище (файловая система или S3), запись URL в БД.

**Результат:** Существующий контент из JSON доступен в платформе; учитель может дополнять его через веб.

---

#### Спринт 2.3 — Кабинеты ученика и учителя

**Цель:** Базовый функционал ролей.

**Задачи:**
- Ученик: список классов/предметов, параграфы, тесты, прохождение теста, сохранение попытки в БД, экран результата.
- Учитель: управление контентом (в рамках своих классов/предметов), создание тестов, в т.ч. генерация по датам/терминам/персонам; просмотр списка учеников и их попыток.
- Админ: управление пользователями, классами, ролями; полный доступ к контенту.

**Результат:** Роли реализованы на уровне экранов и API; попытки сохраняются.

---

#### Спринт 2.4 — Статистика, прогресс, рейтинг

**Цель:** Аналитика и геймификация.

**Задачи:**
- История попыток ученика, баллы по тестам, прогресс по темам (на основе пройденных тестов).
- Рейтинг: по классу, по школе; период (неделя/месяц/всё время).
- Учитель: отчёты по классу, экспорт в CSV/Excel; какие вопросы чаще ошибочные.

**Результат:** Ученики видят свой прогресс и место в рейтинге; учителя — сводки и экспорт.

---

#### Спринт 2.5 — Назначения и расширенная аналитика

**Цель:** Управление обучением и углублённая аналитика.

**Задачи:**
- Назначения: учитель назначает тест классу или группе с дедлайном; ученик видит «домашние тесты» и сдаёт их.
- Достижения (бейджи) за количество пройденных тестов, за 100% по тесту и т.д.
- Расширенная аналитика: время на вопрос, распределение ошибок по вопросам, сравнение классов.

**Результат:** Полноценный цикл «назначил → прошёл → посмотрел статистику»; достижения и углублённые отчёты.

## 6. Стратегия миграции Stage 1 → Stage 2

### 6.1 Сохранение инвестиций Stage 1

- **Единая схема JSON** в Stage 1 совпадает с тем, что будет импортироваться в Stage 2: те же сущности (параграф, раздел, дата, термин, персона, тест, вопрос) и поля (`id`, `difficulty`, `image` и т.д.).
- **Движок тестов** остаётся переиспользуемым: в Stage 2 API отдаёт массив вопросов, фронт (React) вызывает тот же движок проверки и логику попыток; разница только в том, что ответы уходят на сервер и сохраняются в БД.
- **Генераторы тестов** (по датам/терминам/персонам) переносятся на backend: те же правила генерации, вызов из API при создании теста учителем.

### 6.2 Пошаговая миграция

1. **Подготовка данных:** В Stage 1 довести все предметы до единого формата JSON (версия схемы, id везде где нужно, image как путь или base64). Регулярный экспорт из админки как «канонический» бэкап.
2. **Backend и БД:** Реализовать API и таблицы по той же схеме сущностей; скрипт импорта из JSON (и медиа) в БД.
3. **Фронт:** Новое SPA (React) для Stage 2; параллельно можно держать статический портал Stage 1 для офлайн/дома. В новом фронте — те же сценарии (параграф, тест, карточки), но с авторизацией и запросами к API.
4. **Выкладка:** Сначала Stage 2 — внутренний бета-тест (одна школа/класс); обратная связь; затем отказ от старого портала или сохранение его как «офлайн-версия» для домашнего использования.

### 6.3 Риски миграции

| Риск | Митигация |
|------|-----------|
| Расхождение формата JSON и схемы БД | Чёткая документация схемы (JSON Schema); один скрипт импорта с валидацией и откатом. |
| Потеря медиа при переносе | Экспорт из Stage 1 с папкой media или base64; импорт в Stage 2 с загрузкой в хранилище и проставлением URL. |
| Разное поведение движка на старом и новом фронте | Вынести движок в общий пакет (npm) или копировать с тестами; на новом фронте использовать тот же модуль. |

## 7. Риски и сложность (сводка)

### Stage 1

| Область | Риск / сложность | Рекомендация |
|---------|-------------------|--------------|
| Размер JSON при большом контенте | Медленная загрузка, лимиты localStorage | Разбиение по предметам; ленивая загрузка параграфов; при экспорте — разбивать на файлы по предмету. |
| Изображения без сервера | Хранение base64 раздувает JSON | Предпочитать файлы в `media/` и относительные пути; base64 только для маленьких иконок. |
| Drag & Drop в старых браузерах | Не везде одинаковое поведение | Использовать SortableJS с fallback или явно ограничить поддержку браузеров. |
| Генераторы тестов | Разное качество вопросов по предметам | Шаблоны вопросов настраиваемые (например, «Год по событию», «Событие по году»); возможность править сгенерированный тест в админке перед сохранением. |
| Адаптивная сложность | Субъективность easy/medium/hard | Задать простые правила по длине текста и типу вопроса; дать учителю возможность править уровень вручную. |

### Stage 2

| Область | Риск / сложность | Рекомендация |
|---------|-------------------|--------------|
| Рост кодовой базы | Сложность поддержки двух фронтов (старый + новый) | Чётко ограничить срок жизни старого; переносить по модулям (движок, типы). |
| Производительность БД | Большое число попыток и ответов | Индексы по user_id, quiz_id, started_at; пагинация и агрегаты на стороне БД. |
| Безопасность | Доступ к контенту и результатам по ролям | Единый слой авторизации в API (middleware по ролям); не доверять фронту в проверке прав. |

## 8. Дополнительные улучшения (предложения)

1. **Версионирование схемы JSON:** Поле `version` в корне JSON; при импорте и в админке проверять версию и при необходимости запускать миграции (например, добавление полей по умолчанию). Это упростит эволюцию формата и миграцию в Stage 2.
2. **Единый конфиг предмета:** Один файл `content.json` на предмет (paragraphs + quizzes + meta) уменьшает количество запросов и упрощает экспорт/импорт; обратная совместимость — загрузчик умеет и старый формат (paragraphs.json + quiz-*.json), и новый.
3. **Офлайн-статистика в Stage 1:** В localStorage сохранять последние N попыток (id теста, балл, дата) без учётки; показывать на главной «Недавно пройдено» и «Лучший результат по тесту». Это подготовит пользователей к идее статистики в Stage 2.
4. **Шаблоны вопросов для генераторов:** В конфиге или в коде задать шаблоны («В каком году произошло: {event}?», «Какое событие в {year}?») с подстановкой полей из dates/terms/people; так проще добавлять новые типы вопросов и локализовать формулировки.
5. **Режим «экзамен» в Stage 1:** Таймер на весь тест, запрет перехода назад к уже отвеченным вопросам, одна попытка без подсказок — опция при старте теста. Реализуется в движке и UI без бэкенда.
6. **Документация для авторов контента:** Краткий гайд: структура папок, как заполнять JSON вручную (если нужно), как пользоваться админкой, как делать бэкап. Снизит нагрузку на разработку «как добавить предмет».
